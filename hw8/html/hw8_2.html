Problem 2

Until 2002, it was an open question whether the decision problem "Is integer nnn a composite number, in other words, not prime?" could be computed in time polynomial in the size of its input. Why doesn’t the following algorithm suffice to prove it is in P, since it runs in O(n)O(n)O(n) time?

PrimalityTesting(n)composite←falsefor i←2 to n−1if  n mod i=0composite←truereturn composite\text{PrimalityTesting}(n) \\ \qquad \text{composite} \gets \textbf{false} \\ \qquad \textbf{for} \ i \gets 2 \ \text{to} \ n-1 \\ \qquad \qquad \textbf{if } \ n \text{ mod } i = 0 \\ \qquad \qquad \qquad \text{composite} \gets \textbf{true} \\ \qquad \textbf{return} \ \text{composite} PrimalityTesting(n)composite←falsefor i←2 to n−1if  n mod i=0composite←truereturn composite


Solution

This is more or less a word game. The algorithm, despite its correctness, is not polynomial in the size of its input -- in other words, the number of digits (in any base aaa).

We are looking for an algorithm that runs in O(s)O(s)O(s) time for an input of size s=log⁡ns = \log ns=logn, but our intuitive PrimalityTesting\text{PrimalityTesting}PrimalityTesting algorithm actually has O(n)=O(as)O(n) = O(a^s)O(n)=O(as) complexity!

As pointed out by the well-known paper on the AKS Primality Test (that gives an authentic polynomial algorithm),

Let PRIMES\text{PRIMES}PRIMES denote the set of all prime numbers. The definition of prime numbers already gives a way of determining if a number nnn is in PRIMES\text{PRIMES}PRIMES: try dividing nnn by every number m≤nm \le \sqrt{n}m≤n​ -- if any mmm divides nnn then it is composite, otherwise it is prime.
……The test, however, is inefficient: it takes Ω(n)\Omega(\sqrt{n})Ω(n​) steps to determine if nnn is prime. An efficient test should need only a polynomial (in the size of the input = ⌈log⁡n⌉\lceil \log n \rceil⌈logn⌉) number of steps.

Agrawal, M.; Kayal, N.; and Saxena, N. "Primes is in P." Ann. Math. 160, 781-793, 2004.
